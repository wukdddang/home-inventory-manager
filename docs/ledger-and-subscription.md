# 가계부·구독 연동 방안

나중에 **가계부**와 **구독 내역**을 이 프로젝트에 붙일 때 참고할 정리입니다.

---

## 1. 가계부 — 엔티티 추가 방식

지금 구조(Account, AccountTransaction, RecurringIncome)를 **가계부**로 확장하면 됩니다.

### 이미 있는 것

- **Account**: 잔고 계정(현금, 통장)
- **AccountTransaction**: 입출금 이력
- **RecurringIncome**: 예정 수입(월급 등)

### 가계부용으로 추가하면 좋은 엔티티

| 엔티티 | 역할 | 비고 |
|--------|------|------|
| **TransactionCategory** (또는 기존 Category 확장) | 수입/지출 분류 (급여, 식비, 교통비, 통신비 등) | 가계부 전용이면 별도 엔티티, 아니면 Category에 type 추가 |
| **Expense** / **AccountTransaction 확장** | 지출 기록 (날짜, 금액, 결제 수단, 메모, 카테고리) | AccountTransaction에 categoryId, type(income/expense) 넣어도 됨 |
| **RecurringExpense** | 고정 지출 (월세, 관리비, 보험, **구독료** 등) | RecurringIncome과 대칭 |
| **Budget** | 월별·카테고리별 예산 | 예산 대비 지출 알림 |

가계부 화면에서는 **Account + AccountTransaction(수입/지출) + RecurringIncome/RecurringExpense + Budget** 조합으로 “이번 달 수입/지출/잔액”을 보여주면 됩니다.  
→ **결론: 기존 도메인에 엔티티 추가하는 방식으로 가계부까지 커버 가능합니다.**

---

## 2. 구독 내역 — 데이터를 어떻게 가져올까?

사용자가 **구독 이름, 금액, 결제일** 등을 전부 직접 입력하면 부담이 큽니다. 선택지는 대략 아래와 같습니다.

### 2-1. 수동 입력 + 알림 (가장 단순)

- 사용자가 구독을 **한 번만 등록** (이름, 금액, 결제 주기, 다음 결제일).
- **RecurringExpense** 또는 **Subscription** 엔티티로 저장.
- 다음 결제일 N일 전에 **알림**만 보내기 (Notification).
- 장점: 구현 쉬움, 외부 API 없음.  
- 단점: 등록·갱신을 사용자가 직접 해야 함.

→ “다른 방법이 없으면 이걸로 가야지” 하신 것처럼, **기본값**으로 두고, 아래 방식들은 “나중에 편의 개선”으로 붙이는 게 좋습니다.

---

### 2-2. 오픈뱅킹 / 금융 데이터 연동 (자동·반자동)

**은행/카드 거래 내역**을 가져와서, 그중에서 **반복 결제(구독)** 를 찾는 방식입니다.

- **한국**
  - **오픈뱅킹 API**: 금융결제원 등에서 제공. 사용자 동의 후 계좌·거래 내역 조회 가능.  
    → 거래 목록을 가져와서 “매월 비슷한 금액·비슷한 일자”로 **구독 후보**를 추천하고, 사용자가 “이거 구독이야”만 체크하게 할 수 있음.
  - **뱅크샐러드·토스 등**: 개인 개발자가 직접 연동하기는 어렵고, 해당 서비스가 제공하는 API가 있으면 그걸 쓰는 수밖에 없음. (일반적으로는 없음.)
- **해외**
  - **Plaid**(미국 등): 은행/카드 연결 후 거래 내역 조회. 구독 탐지 기능을 제공하는 경우도 있음.
  - **Yodlee** 등 다른 aggregation 서비스도 비슷한 역할.

**정리**:  
- 국내는 **오픈뱅킹**으로 “거래 내역 가져오기 → 반복 결제 후보 추출 → 사용자가 구독으로 등록” 플로우가 현실적입니다.  
- “완전 자동으로 구독 목록 채우기”보다는 **“거래에서 후보를 추천해 주고, 사용자는 확인/등록만”** 하면 입력 부담이 많이 줄어듭니다.

---

### 2-3. 이메일에서 구독 정보 파싱 (반자동)

넷플릭스, 유튜브 프리미엄, 멤버십 등 **결제 확인 메일**이 오는 경우가 많습니다.

- **Gmail API** 등으로 메일 수신 → “결제 완료”, “다음 결제일”, “금액” 같은 문구/날짜를 파싱.
- 파싱 결과로 **구독 후보** 또는 **실제 결제 이력**을 만들어 DB에 넣고, 사용자는 “이 구독 등록할까요?”만 선택.

**고려할 점**

- 개인정보·보안: 이메일 접근 권한이 크므로, **사용자 동의**와 **최소 권한**(구독 관련 메일만 등) 설계가 필요함.
- 파싱 난이도: 서비스마다 메일 형식이 달라서, 주요 서비스(넷플릭스, 스포티파이 등)부터 규칙을 만드는 식으로 단계적으로 확장하는 게 좋음.
- 해외 서비스는 영어 메일이라 규칙 만들기는 상대적으로 수월함.

→ “직접 다 입력하기 귀찮다”를 줄이는 **보조 수단**으로 쓰기 좋고, “이메일 연동”을 켜면 구독 후보를 자동으로 채워 주는 식으로 설계할 수 있습니다.

---

### 2-4. 구독 전용 서비스 API (있다면)

- 해외에는 **Truebill**, **Bobby** 같은 구독 관리 앱이 있지만, 개인 프로젝트에서 API를 쓰기엔 제한이 있을 수 있음.
- 국내에 “구독 목록만 공개 API로 주는 서비스”는 잘 없음.  
→ 당장 설계 목표로 두기보다는, 나중에 이런 API가 생기면 **연동 레이어**만 만들어 두는 정도가 맞습니다.

---

### 2-5. 정리: 구독 데이터 “다른 방법” 요약

| 방법 | 난이도 | 사용자 입력 | 비고 |
|------|--------|-------------|------|
| **수동 입력 + 다음 결제일 알림** | 낮음 | 매번 아님, 최초 1회 등록 | 기본값으로 추천 |
| **오픈뱅킹 → 거래 내역 → 반복 결제 후보 추천** | 중간 | “이거 구독이야” 확인 정도 | 국내에서 가장 현실적인 자동화 |
| **이메일 파싱(Gmail API 등) → 구독 후보 자동 생성** | 중간~높음 | 확인/등록만 | 보안·동의 설계 필요 |
| **외부 구독 관리 API** | API 존재 시 | 거의 없음 | 국내는 옵션 거의 없음 |

**추천 흐름**

1. **1단계**: RecurringExpense(또는 Subscription) + 수동 입력 + “다음 결제일 알림”. (지금 설계만 해두고, 가계부 붙일 때 구현.)
2. **2단계**: 오픈뱅킹(또는 Plaid 등)으로 거래 가져오기 → 반복 결제 후보 추출 → 사용자가 “구독으로 등록”만 하기.
3. **3단계**(선택): 이메일 파싱으로 구독/결제 메일만 읽어서 후보 자동 생성.

“방법이 없다면 입력해야겠지만”이시라면, **1단계를 기본**으로 두고, **2단계(오픈뱅킹/거래 기반)** 가 “귀찮음 줄이는 다른 방법”으로 가장 현실적입니다. 이메일은 보안·설계 부담이 있어서 그다음 단계로 두면 됩니다.

---

## 3. 이 문서와 다른 문서 연결

- **가계부 엔티티**: [considerations.md](./considerations.md)의 “재무·예산” (고정 지출, 예산, 수입/지출 카테고리) + 위 1절.
- **구독 엔티티**: RecurringExpense 확장 또는 **Subscription** (구독명, 금액, 결제 주기, 다음 결제일, 연동 출처: manual / bank / email 등) 으로 두고, 나중에 “연동 출처”별로 데이터 채우는 로직만 추가하면 됩니다.

필요하면 개념적/논리적 설계에 **Subscription**, **RecurringExpense**를 반영해 두면, 가계부·구독 기능 붙일 때 일관되게 확장하기 좋습니다.
