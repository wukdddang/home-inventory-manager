# 집 구조도(3D) 등록 및 물품 배치 기능 — 기술 요구사항

집 구조가 없는 경우 프론트에서 3D 집을 구축하고, 서버 DB에 저장한 뒤, 나중에 해당 3D 모델 위에 배치한 물품을 확인할 수 있게 하는 기능에 필요한 기술을 정리한 문서입니다.

---

## 1. 요구사항 요약

| 구분 | 내용 |
|------|------|
| **입력** | 집 구조도가 없을 때 → 프론트에서 3D 형태의 집(방, 벽, 구역) 구축 |
| **저장** | 구축한 3D 집 모델을 서버 DB에 저장 |
| **조회** | 사용자가 접속 시 저장된 집 모델을 불러와, 그 위에 배치해 둔 물품을 시각적으로 확인 |

---

## 2. 추가로 필요한 기술 요소

### 2.1 프론트엔드 — 3D 렌더링 및 편집

| 기술 | 용도 | 비고 |
|------|------|------|
| **Three.js** | 웹에서 3D 씬 렌더링 (WebGL) | 사실상 표준, 문서/예제 풍부 |
| **React Three Fiber (R3F)** | React 컴포넌트로 Three.js 씬 구성 | React 사용 시 권장 |
| **@react-three/drei** | R3F용 헬퍼(카메라, 컨트롤, 그리드 등) | 3D UI/편집기 구현 시 유용 |
| **@react-three/fiber** | R3F 코어 | — |

**역할**

- **집 편집 모드**: 방/벽/바닥을 박스(BoxGeometry) 등으로 배치해 “집 구조” 정의
- **뷰 모드**: 저장된 구조 데이터를 불러와 3D로 렌더링하고, 물품 마커/레이블 표시

**대안**

- **Babylon.js**: Three.js 대안, 게임/복잡한 인터랙션에 강함
- **A-Frame**: HTML 선언형 3D, 프로토타입에 적합

---

### 2.2 3D 데이터 포맷 — 무엇을 DB에 저장할지

3D “집”을 서버에 저장하려면 **구조를 표현하는 데이터 포맷**이 필요합니다.

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **자체 JSON 스키마** | 방(room), 벽(wall), 바닥(floor), 배치 슬롯(slot) 등을 JSON으로 정의 | DB 저장/쿼리 쉬움, 가벼움, Household/StorageLocation와 연계 용이 | 3D 툴과 호환성 없음 |
| **glTF/GLB** | 표준 3D 포맷 (메쉬, 머티리얼, 씬 그래프) | 다양한 툴 호환, Three.js 기본 지원 | 바이너리/용량 증가, “방/슬롯” 같은 논리 구조는 별도 메타데이터 필요 |

**권장**

- **1단계**: **자체 JSON 스키마**로 “논리적 집 구조”를 저장  
  - 예: `{ rooms: [{ id, name, bounds, floorZ }], slots: [{ id, roomId, position, storageLocationId }] }`
- **2단계(선택)**: 나중에 복잡한 메쉬가 필요하면 glTF 내보내기/가져오기를 추가하고, “논리 구조”는 기존 JSON으로 유지

---

### 2.3 백엔드 — DB 엔티티 및 API

현재 **StorageLocation**(보관 장소)이 Household에 연결되어 있으므로, “집 구조”와 “장소”를 연결하는 엔티티를 추가하는 방식이 자연스럽습니다.

| 추가/변경 엔티티 | 역할 |
|------------------|------|
| **HouseStructure** (또는 FloorPlan) | Household당 1개. “3D 집” 한 채를 나타냄. 구조 데이터(JSON) 저장. |
| **HouseStructureRoom** (선택) | 방 단위 구역. HouseStructure 1:N. 이름, 바운드(위치/크기) 등. |
| **StorageLocation 확장** | 기존 엔티티에 `positionX, positionY, positionZ` 또는 `slotId`(HouseStructure 내 슬롯 참조) 추가 → “어느 방/어느 위치에 있는지” 연결 |

**HouseStructure 예시 필드**

- `householdId` (FK)
- `name` (예: "우리 집")
- `structurePayload` (JSONB): 위에서 정의한 자체 JSON 스키마
- `version` (스키마 버전 관리용)

**API**

- `POST /households/:id/structure` — 구조 생성/덮어쓰기
- `GET /households/:id/structure` — 구조 조회 (프론트에서 3D 복원)
- (선택) `PATCH` 로 부분 수정

---

### 2.4 물품–위치 연결 (배치 확인)

“배치해 둔 물품”을 3D 위에서 보려면:

- **StorageLocation**이 “집 구조 내 한 지점/슬롯”과 연결되어 있어야 함.
- **InventoryItem**은 이미 `StorageLocation`을 갖고 있으므로,  
  “StorageLocation ↔ 3D 위치”만 정해지면, “해당 위치에 어떤 물품이 있는지”는 기존 모델로 조회 가능.

구현 옵션:

1. **StorageLocation에 3D 좌표 추가**  
   - `structureId`, `positionX, positionY, positionZ` (또는 `roomId`, `slotId`)  
   - 이 장소가 “어느 집 구조의 어디”인지 지정.
2. **별도 엔티티 ItemPlacement**  
   - `houseStructureId`, `storageLocationId`, `positionX/Y/Z`  
   - 한 StorageLocation을 여러 “표시 위치”에 매핑할 때 유연함.

초기에는 **1번(StorageLocation 확장)**으로 시작하고, 필요 시 2번으로 분리해도 됩니다.

## 3. 2D 집 구조 — 사용자가 만드는 방식

**빈 캔버스에 선을 그려서 전부 직접 그리게 할 필요는 없습니다.** 아래처럼 하면 부담을 많이 줄일 수 있습니다.

### 3.1 추천: 블록(방) 추가 방식

사용자가 **“방”을 하나씩 추가**하는 방식입니다. 연필/선 그리기가 아니라 **도형 배치**에 가깝습니다.

| 사용자 행동 | 동작 |
|-------------|------|
| **「방 추가」 버튼 클릭** | 기본 크기의 사각형(방)이 캔버스에 하나 생김 |
| **방 이름 입력** | "거실", "주방", "냉장고 옆" 등 |
| **방 드래그** | 위치 이동 |
| **모서리/가장자리 드래그** | 크기 조절 (리사이즈 핸들) |
| **보관 위치(슬롯) 추가** | 방 안에 "여기가 냉장고 문쪽" 같은 포인트/영역 지정 → 기존 StorageLocation과 연결 |

- **그리는 것 없음**: 선/벽을 일일이 그리지 않고, **방 = 사각형 한 개**로 두고 필요한 만큼 추가·이동·리사이즈만 하면 됨.
- 데이터는 `rooms: [{ id, name, x, y, width, height }], slots: [...]` 형태의 JSON으로 저장 가능 (2D면 z 불필요).

### 3.2 선택: 템플릿에서 시작

처음부터 빈 화면이 아니라 **자주 쓰는 평면 유형**을 골라서 시작하게 할 수 있습니다.

- 예: "원룸(거실+주방 한 공간)", "2룸(거실+방1+방2)", "주방 분리형 3룸" 등 3~5가지.
- 사용자는 **템플릿 선택 → 방 이름만 수정하거나 방 크기만 살짝 조정**하면 됨.
- 구현: 템플릿별로 미리 정의한 `structurePayload` JSON을 넣어 두고, 선택 시 그걸 로드한 뒤 편집 가능하게 함.

### 3.3 선택: 평면도 이미지 올리고 위치만 찍기

집 평면도/도면 이미지가 있는 사용자를 위한 방식입니다.

- 사용자가 **이미지 파일 업로드** (예: 집 배치도, 간단한 스케치).
- 화면에는 그 이미지가 배경으로 보이고, 사용자는 **“여기가 냉장고”, “여기가 욕실장”**처럼 **위치(마커/핀)만 클릭해서 찍고** 기존 StorageLocation과 연결.
- “구조를 그리는” 작업은 없고, **이미 있는 그림 위에 보관 위치만 매핑**하는 형태.

---

### 정리: 2D에서 사용자 부담 줄이기

| 방식 | 사용자 할 일 | 구현 난이도 |
|------|--------------|-------------|
| **블록(방) 추가** | 방 추가 → 이름 입력 → 위치/크기 조절 → (선택) 슬롯 연결 | 낮음, 권장 |
| **템플릿에서 시작** | 템플릿 선택 → 이름/크기만 수정 | 낮음 (템플릿 JSON만 준비) |
| **이미지 + 위치 찍기** | 평면도 이미지 업로드 → 마커로 보관 위치만 지정 | 중간 (이미지 저장/표시 필요) |
| ~~빈 화면에 전부 직접 그리기~~ | 선/벽 직접 그리기 | 높음, 비권장 |

**결론:** 2D라도 **“빈 화면에 스스로 모든 구조도를 다 그리게” 하지 말고**,  
**「방(사각형)을 추가하고 이름·위치·크기만 조절」**하는 블록 방식(+ 필요 시 템플릿)으로 가면 만들기 쉽고 사용자도 부담이 적습니다.

### 3.4 확정 스코프: ERD 스타일 + 이름 변경 + 방 클릭 시 물품 정보

- **표현**: 큰방, 작은방1, 작은방2, 화장실1 처럼 **방 단위 박스 + 라벨**만 표시 (ERD처럼).
- **편집**: **이름 변경만** 허용 (위치/크기는 템플릿 또는 기본값 고정해도 됨).
- **연동**: 각 "방"은 하나의 **StorageLocation**(또는 방:장소 1:N)과 연결.
- **UX**: 프론트에서 **해당 방을 클릭하면** → 그 방(StorageLocation)에 연결된 **물품 목록/정보**를 패널이나 모달로 표시.

이 스코프면 "그리는" 기능은 최소화되고, **박스 목록 + 이름 수정 + 클릭 시 상세**만 구현하면 됨.

## 4. 2D 구현용 라이브러리 추천

ERD처럼 **방 = 박스 + 라벨**, **이름 변경 + 방 클릭 → 물품 정보** 수준이라면 아래 중에서 선택하면 됩니다.

| 라이브러리 | 방식 | 장점 | 단점 | 적합도 |
|------------|------|------|------|--------|
| **라이브러리 없음 (SVG + React)** | `<svg>` 안에 `<rect>` + `<text>` (또는 `<foreignObject>`로 라벨), state로 `rooms[]` 관리 | 의존성 없음, 구현 단순, 번들 작음 | 드래그/리사이즈는 직접 구현 | ★★★★★ (이 스코프에 가장 적합) |
| **react-konva** | Canvas 기반, `Stage` > `Layer` > `Rect`, `Text`, `Group` | 드래그·리사이즈·클릭 내장, 문서 양호 | Canvas라서 DOM 검사/접근성은 SVG보다 불리 | ★★★★ |
| **React Flow (@xyflow/react)** | 방을 Node로 두고, Edge 없이 사용 | 줌/팬 기본, 노드 클릭/선택 내장 | "플로우" 개념과는 다름, 노드 커스터마이징 필요 | ★★★ |
| **Fabric.js (react-fabric)** | Canvas, 도형·텍스트·이벤트 | 편집기형 UI에 강함 | 무겁고, React와의 결합은 보조 라이브러리 필요 | ★★ |

### 권장안

- **현재 스코프(이름 변경 + 방 클릭 → 물품 정보, 그리기는 최소)**  
  → **라이브러리 없이 SVG + React**로 구현하는 것을 권장합니다.  
  - `rooms: [{ id, name, x, y, width, height }]` 상태만 두고,  
  - 각 방을 `<rect>` + 라벨로 그린 뒤 **클릭** 시 선택 상태 저장 → 우측/하단 패널에 해당 방의 StorageLocation 기준 **물품 목록** 표시.  
  - **이름 변경**은 방 더블클릭 시 인라인 `<input>` 또는 작은 모달로 처리하면 됩니다.
- **나중에 "방 드래그로 배치 변경"까지 넣고 싶다면**  
  → **react-konva**를 도입해 드래그(및 선택적 리사이즈)만 추가하는 방식이 무난합니다.

## 6. 구현 순서 제안

1. **백엔드**  
   - `HouseStructure` 엔티티 추가 (householdId, name, structurePayload JSONB).  
   - CRUD API (생성/조회/수정).
2. **프론트 (뷰어)**  
   - Three.js(R3F)로 `structurePayload`만 받아서 “방/벽/슬롯”을 박스로 그리기.  
   - 카메라/조명/간단한 컨트롤(drei).
3. **프론트 (편집기)**  
   - 방/벽 추가·이동·크기 조절 → `structurePayload` 생성 → API로 저장.
4. **StorageLocation ↔ 3D**  
   - StorageLocation에 `structureId` + 위치(slotId 또는 x,y,z) 추가.  
   - 3D 뷰에서 “이 슬롯 = 이 StorageLocation” 표시 후, 해당 장소의 InventoryItem 목록 표시.

이 순서로 진행하면 “집 구조 없을 때 3D로 구축 → DB 저장 → 나중에 모델 보면서 배치한 물품 확인” 플로우를 단계적으로 구현할 수 있습니다.
